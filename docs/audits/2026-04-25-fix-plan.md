## 3. תיקוני אלגוריתם SRS / בחירת שאלות (Q) — לב הבעיה

### הבנת ה-flow הנוכחי

```
1. SetupView.handleStart →
   1.1 fetchSrsData()   ← 🔴 קורא רק next_review_date
   1.2 selectSmartQuestions(pool, count, sessionSize, srsData, history, allData)

2. selectSmartQuestions (smartSelection.ts):
   2.1 computeTopicStats(history, data) — חישוב סטטיסטיקות פר נושא
   2.2 STAGE 1: computeTopicScores — מחשב ציון ל-נושא (0.5 default ל-weakness, 1.0 ל-recency)
   2.3 STAGE 2: allocateSlots (Hamilton) — חלוקת מקומות פר נושא
   2.4 STAGE 3: בתוך כל נושא → 🔴 רק computeSrsUrgency + Math.random*0.001
   2.5 Fill gaps → 🔴 שוב רק computeSrsUrgency + Math.random*0.001
   2.6 Final shuffle (זה תקין — לסדר התצוגה)
```

**התיקון:**

### Q3 — `computeSrsUrgency` חסר עדיפות שלילית לעתיד 🟠

**הבעיה הנוכחית** ([smartSelection.ts:166-172](src/lib/smartSelection.ts#L166-L172)):
```typescript
function computeSrsUrgency(q, srsData) {
  const srs = srsData[q[KEYS.ID]];
  if (!srs) return 0.5;          // אין רשומה → בינוני
  const daysOverdue = ...;
  if (daysOverdue <= 0) return 0;   // 🔴 לא הגיע תורה → אפס
  return clamp01(daysOverdue / 60);
}
```

תוצאה:
- שאלה שתאריך החזרה שלה היום: urgency = 0
- שאלה שתאריך החזרה שלה עוד שנה: urgency = 0
- שאלה חדשה לחלוטין: urgency = 0.5 (יותר!)

**שאלה חדשה מנצחת שאלה ש-SM-2 קבע שצריך לחזור עליה רק עוד שנה.** זה תקין כשרק יש "חדשות". אבל כש-pool מלא בשאלות שתוזמנו לעתיד, **ה-Math.random הקטן הופך לקובע היחיד.**

**התיקון** — החזר ערכים שליליים לעתיד הרחוק כדי לדכא בחירה:
```typescript
function computeSrsUrgency(q, srsData) {
  const srs = srsData[q[KEYS.ID]];
  if (!srs) return 0.5;
  const daysOverdue = ...;
  if (daysOverdue >= 0) return clamp01(daysOverdue / 60);
  // עתיד: דיכוי יחסי לכמה רחוק. -1 על שנה, -0.1 על 10 ימים.
  return Math.max(-1, daysOverdue / 365);
}
```

**ההשפעה:** שאלות שתוזמנו לעתיד יורדות מתחת ל-0; שאלות חדשות (0.5) ושאלות שעבר זמנן (>0) יקבלו עדיפות. אם יש מעט-מאוד-מאוד שאלות "ראויות" — ה-fill gaps ייקח את הפחות-עתידיות.

**סיכון בתיקון:** 🟡 בינוני — שינוי תפיסת ערך urgency. יצריך adjustment ב-`computeSmartScore` כדי שלא יהפך ה-SRS לשולי.

### Q4 — הסר את `Math.random() * 0.001` 🔴

**הבעיה:** [smartSelection.ts:422 ו-432](src/lib/smartSelection.ts#L422). שאלות ששוות ב-urgency (כל ה-future-scheduled מקבלות 0) — נבחרות **אקראית** ב-tiebreaker.

**התיקון:** החלפה ב-tiebreaker דטרמיניסטי. הצעה: זמן עדכון אחרון של ה-SRS record. בלי זה — שעון פנימי או question.id (alphabetical) עובד גם.

```typescript
// במקום: + Math.random() * 0.001
// כש-Q1 מוטמע, השדה score כולל את כל הגורמים. ה-tiebreaker הופך לסטייבל:
.sort((a, b) => {
  const diff = b.score - a.score;
  if (diff !== 0) return diff;
  // tiebreaker: שאלה שלא נראתה הכי הרבה זמן — ראשונה
  const aTs = history[a.q[KEYS.ID]]?.timestamp ?? 0;
  const bTs = history[b.q[KEYS.ID]]?.timestamp ?? 0;
  return aTs - bTs;
});
```

**ההשפעה:** מאותם 100 שאלות "בטוח" — הראשונה תהיה זו שלא נראתה הכי הרבה זמן (ולא רנדומלית). דטרמיניסטי, צפוי, ניתן לבדיקה ב-unit test.

**סיכון בתיקון:** 🟢 נמוך.

## 6. ⚠️ בונוס: שדרוג ל-FSRS (אופציונלי, רק באישור משתמש)

**FSRS** (Free Spaced Repetition Scheduler) — אלגוריתם שהחליף את SM-2 ב-Anki ב-2023. מבוסס מחקר אקדמי, מותאם ב-19 hyperparameters. דיוק 30-50% טוב יותר מ-SM-2 בחיזוי שכחה.

### האם זה אפשרי? **כן, מבחינה טכנית.**

| גורם | מצב |
|---|---|
| Library | `ts-fsrs` ב-npm (TypeScript-native, MIT license, ~10KB minified) |
| API | `f.next(card, now, rating)` → מחזיר card מעודכן + scheduling |
| Rating mapping | guessed→Again(1), hesitant→Hard(2), confident→Good(3) — מתאים לקיים |
| Bundle impact | זניח (10KB מתוך bundle של ~500KB) |

### האם זה מסוכן? **בינוני.**

**סיכונים אמיתיים:**

1. **🟠 Migration של ה-DB.** FSRS דורש שדות חדשים: `stability`, `difficulty`, `elapsed_days`, `scheduled_days`, `state`, `lapses`, `last_review`. צריך:
   - Migration שמוסיף עמודות (לא הורס נתונים — `IF NOT EXISTS`)
   - אסטרטגיית backfill: או "fresh start" (איפוס SRS לכל המשתמשים) או חישוב היוריסטי מהנתונים הקיימים
   - **לא קל. לא טריוויאלי. אבל אפשרי.**

2. **🟠 שינוי התנהגות נראה למשתמש.** שאלה שהיתה אמורה לחזור בעוד 5 ימים לפי SM-2 — אולי תחזור בעוד 8 ימים לפי FSRS. תקופת הסתגלות. צריך תקשורת למשתמש.
